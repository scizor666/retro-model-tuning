# GGUF Export and Ollama Integration

This document outlines the process of exporting the fine-tuned Gemma-3 model to GGUF format and serving it via Ollama, including the specific fixes required for Gemma-3's unique architecture.

## 🧰 Required Tools
- **llama.cpp**: Custom build with Gemma-3 patches (located at `/mnt/ntfs/llama.cpp`).
- **Unsloth**: Used for robust weight merging and dequantization.
- **Ollama**: Local LLM runner.

## 🔧 Core Fixes for Gemma-3

### 1. Tokenizer Support
Gemma-3 uses a specific BPE pre-tokenizer. We patched `convert_hf_to_gguf.py` to include its hash:
- **File**: `llama.cpp/convert_hf_to_gguf.py`
- **Hash**: `789696f5946cc0fc59371f39f6097cafed196b3acded6140432f26bbb1ae1669`
- **Mapping**: Added to the `chkhsh` checks to map to `llama-bpe`.

### 2. Multimodal Tensor Filtering
Gemma-3-4B-IT has vision components. For text-only Ollama use, we must skip these:
- **Fix**: Updated `Gemma3Model.modify_tensors` in `convert_hf_to_gguf.py` to use `in` instead of `startswith` when checking for `multi_modal_projector` and `vision_tower`. This handles the `language_model.model` prefix generated by Unsloth.

## 🚀 Export Workflow

### Step 1: Merge and Dequantize
Use [merge_export_unsloth.py](file:///home/alex/projects/retro-model-tuning/training/merge_export_unsloth.py). This script:
1. Loads the 4-bit LoRA model.
2. Uses Unsloth's `save_pretrained_gguf` to handle the heavy lifting of merging and dequantizing weights to Float16.
3. Outputs merged safetensors with correct shapes to `/mnt/ntfs/gemma3-retro-export`.

### Step 2: Convert to GGUF
Run the patched conversion script:
```bash
python /mnt/ntfs/llama.cpp/convert_hf_to_gguf.py \
  /mnt/ntfs/gemma3-retro-export \
  --outtype f16 \
  --outfile /mnt/ntfs/gemma3-retro-v4-f16.gguf
```

### Step 3: Quantize
Use the built binary from the `llama.cpp` repo:
```bash
/mnt/ntfs/llama.cpp/build/bin/llama-quantize \
  /mnt/ntfs/gemma3-retro-v4-f16.gguf \
  /mnt/ntfs/gemma3-retro-v4-Q4_K_M.gguf \
  Q4_K_M
```

## 🐋 Ollama Integration

### Storage Redirection (Crucial for sda1 space)
If `sda1` is full, run Ollama as a user process directed to the NVMe drive:
```bash
export OLLAMA_MODELS=/mnt/ntfs/ollama_models
export OLLAMA_HOST=127.0.0.1:11435
ollama serve &
```

### Import Model
Create a `Modelfile` (e.g., `/mnt/ntfs/Modelfile-retro-v4`):
```dockerfile
FROM /mnt/ntfs/gemma3-retro-v4-Q4_K_M.gguf
SYSTEM "You are a helpful retro gaming assistant..."
PARAMETER temperature 0.7
```
Then import:
```bash
OLLAMA_HOST=127.0.0.1:11435 ollama create retro-assistant -f /mnt/ntfs/Modelfile-retro-v4
```
